## 1. Принципы архитектуры

Все решения принимаются исходя из следующих принципов:

| Принцип | Что это означает на практике |
|---|---|
| **Модульность** | Каждый компонент независим — его можно заменить, отключить или обновить не трогая остальные |
| **Расширяемость** | Новый инструмент или агент подключается без переписывания системы |
| **Отказоустойчивость** | При сбое одного инструмента система автоматически переключается на альтернативу |
| **Наблюдаемость** | Каждый шаг логируется — понятно что происходит, где сбой, сколько стоит |
| **Масштабируемость** | Архитектура выдерживает рост нагрузки — от 1 пользователя до сотен |
| **Простота обновления** | Выход нового инструмента = подключение через реестр, без остановки системы |

## 3. Контейнеризация (Docker)

### 3.1 Принцип
Каждый компонент системы запускается в **отдельном Docker-контейнере**. Это обеспечивает:
- Независимость компонентов друг от друга
- Простую замену и обновление отдельных частей
- Одинаковое поведение на любом сервере (dev / staging / prod)
- Горизонтальное масштабирование под нагрузкой

### 3.2 Контейнеры системы

```
docker-compose.yml
│
├── 🤖 orchestrator          # Оркестратор (Open Router + логика)
├── 🔍 agent-scout           # Агент разведчик (Агент 1)
├── 🎬 agent-transcriber     # Агент транскрибации (Агент 2)
├── 🌐 mcp-apify             # MCP-сервер Apify
├── 🕸️  mcp-firecrawl         # MCP-сервер Firecrawl
├── 🔎 mcp-perplexity        # MCP-сервер Perplexity
├── 🎙️  mcp-whisper           # MCP-сервер Whisper
├── 🤖 telegram-bot          # Telegram Bot Gateway
├── 🗄️  database              # База данных (Supabase / PostgreSQL)
├── 📋 scheduler             # Планировщик задач (cron / n8n)
└── 📊 logger                # Логирование и метрики

[ + ] — новые контейнеры при добавлении инструментов
```

### 3.3 Сеть контейнеров
- Все контейнеры находятся в одной внутренней Docker-сети
- Наружу открыт только Telegram Bot Gateway
- Межсервисное общение — через внутренние порты и MCP-протокол

## 4. MCP-протокол (межагентное взаимодействие)

### 4.1 Что такое MCP в данном проекте
MCP (Model Context Protocol) — стандартный протокол для общения между агентами, оркестраторами и инструментами. Используется как единый язык подключения компонентов системы.

### 4.2 Схема MCP-подключений

```
[Главный оркестратор Marketing agency]
              ↕ MCP
[Оркестратор Агента-разведчика]
              ↕ MCP
    ┌─────────┼──────────┐
    ↕         ↕          ↕
[Агент 1] [Агент 2]  [Агент N...]
    ↕         ↕          ↕
[MCP Tool Registry — реестр инструментов]
    ↕         ↕          ↕
[Apify]  [Firecrawl] [Perplexity] [Whisper] [...]
```

### 4.3 Подключение к внешним агентам и проектам
- Агент-разведчик экспортирует **MCP-интерфейс** — любой внешний оркестратор может его вызвать
- Агент может быть подключён к другим агентам проекта Marketing agency через тот же протокол
- Новые агенты подключаются к реестру без изменения существующего кода

## 5. Реестр инструментов (Tool Registry)

### 5.1 Концепция
Все внешние инструменты (Apify, Firecrawl, Perplexity и др.) регистрируются в **едином реестре**. Оркестратор не обращается к инструментам напрямую — он спрашивает реестр: "кто умеет парсить соц. сети?" и получает список доступных вариантов.

### 5.2 Структура записи в реестре

```yaml
# Пример записи инструмента в реестре
tool:
  id: apify-social-parser
  name: Apify Social Media Parser
  version: "2.1.0"
  capabilities:
    - parse_social_media
    - find_trending_videos
    - scrape_tiktok
    - scrape_youtube
  status: active          # active / degraded / offline
  priority: 1             # чем ниже — тем выше приоритет
  cost_per_call: 0.05     # USD за вызов
  fallback:
    - bright-data-parser  # альтернатива при сбое
    - crawlee-scraper
  mcp_endpoint: "mcp://mcp-apify:3001"

[ + ] — новые поля записи инструмента
```

### 5.3 Как добавляется новый инструмент
1. Создать MCP-сервер для инструмента (или использовать готовый из Smithery)
2. Добавить контейнер в `docker-compose.yml`
3. Зарегистрировать запись в реестре инструментов
4. Описать `capabilities` — что умеет инструмент
5. Указать `fallback` — на что переключиться при сбое
6. Перезапускать систему **не нужно** — реестр обновляется горячо

## 6. Отказоустойчивость и взаимозаменяемость

### 6.1 Механизм Fallback

```
Оркестратор вызывает инструмент
         ↓
  Инструмент недоступен?
         ↓ да
  Смотрим список fallback в реестре
         ↓
  Переключаемся на следующий по приоритету
         ↓
  Логируем факт переключения
         ↓
  Уведомляем пользователя (опционально)
         ↓
  Выполняем задачу на альтернативном инструменте
```

### 6.2 Уровни деградации

| Уровень | Ситуация | Поведение системы |
|---|---|---|
| **Нормальный** | Все инструменты работают | Стандартная работа |
| **Деградированный** | 1–2 инструмента недоступны | Fallback, пониженное качество, уведомление |
| **Аварийный** | Критические инструменты недоступны | Частичный отчёт с пометкой о неполноте |
| **Полный сбой** | Оркестратор недоступен | Уведомление пользователю, ожидание восстановления |

### 6.3 Health Check (проверка здоровья)
- Каждый инструмент проверяется каждые **5 минут**
- Статус в реестре обновляется автоматически: `active / degraded / offline`
- Оркестратор использует только `active` инструменты

## 7. Обновление инструментов и компонентов

### 7.1 Принцип горячего обновления
Система спроектирована так, чтобы **обновление одного компонента не останавливало работу остальных**.

### 7.2 Алгоритм обновления инструмента

```
1. Выходит новая версия инструмента (например, Apify v3)
         ↓
2. Создаём новый контейнер с новой версией рядом со старым
         ↓
3. Регистрируем новую версию в реестре с низким приоритетом
         ↓
4. Тестируем новую версию на малом % запросов
         ↓
5. Если всё ок — повышаем приоритет новой версии
         ↓
6. Старая версия уходит в резерв (потом удаляется)
```

### 7.3 Версионирование
- Каждый инструмент имеет версию в реестре
- Несколько версий одного инструмента могут сосуществовать
- Оркестратор выбирает версию по приоритету и статусу

### 7.4 Обновление через Smithery
- Smithery регулярно сканирует доступные MCP-инструменты
- При появлении нового релевантного инструмента — предлагает его к установке
- Установка = добавление контейнера + запись в реестр

## 8. Масштабируемость и нагрузка

### 8.1 Профили нагрузки

| **Средний масштаб** | 50–500 | 250–2500 | 2–3 сервера, балансировщик нагрузки |
| **Большой масштаб** | 500+ | 2500+ | Kubernetes / облачный auto-scaling |

### 8.2 Узкие места и их решения

| Компонент | Риск нагрузки | Решение |
|---|---|---|
| **Оркестратор** | Очередь запросов при пике | Очередь задач (Redis / BullMQ) |
| **Apify** | Лимиты API при большом числе запросов | Кэширование результатов, батчинг |

| **Chat GPT 5.5** | Rate limits OpenAI | Очередь + fallback модели через Open Router |
| **Telegram Bot** | Flood limits Telegram | Пакетная отправка, rate limiter |
| **БД** | Медленные запросы при росте истории | Индексы, партиционирование, архивирование |

## 9. Поток данных (Data Flow)

### 9.1 Полный цикл: от запроса до отчёта

```
[Пользователь]
    ↓ команда в Telegram
[Telegram Bot Gateway]
    ↓ задача в очередь
[Планировщик / Очередь задач]
    ↓ задача с приоритетом
[Оркестратор — Open Router + Chat GPT 5.5]
    ↓ читает Алгоритм работы
    ↓ выбирает инструменты из реестра
    ├──→ [Apify] → сырые данные соц. сетей
    ├──→ [Firecrawl] → текстовый контент сайтов
    └──→ [Perplexity] → проверенные факты
         ↓
    [Агент разведчик] — агрегирует сырые данные
         ↓
    [Агент обработчик]
         ├──→ [Whisper] → транскрипт видео
         └──→ [LLM GPT 5.5] → анализ смыслов и хуков
         ↓
    [Оркестратор — финальный разбор]
    Chat GPT 5.5 формирует отчёт
         ↓
    [Хранилище] — сохранение отчёта в историю
         ↓
    [Telegram Bot] — отправка отчёта пользователю
```

### 9.2 Формат данных между компонентами

```json
// Входящий запрос в оркестратор
{
  "task_id": "uuid",
  "user_id": "telegram_user_id",
  "type": "search",          // search | create_content | report
  "topics": ["крипта", "маркетинг"],
  "platforms": ["telegram", "youtube", "tiktok"],
  "language": "ru",
  "depth": "standard"        // quick | standard | deep
}

// Исходящий результат от оркестратора
{
  "task_id": "uuid",
  "status": "completed",
  "report": {
    "trends": [...],
    "by_platform": {...},
    "content_ideas": [...],
    "alerts": [...]
  },
  "meta": {
    "tools_used": ["apify", "whisper", "perplexity"],
    "duration_sec": 187,
    "cost_usd": 0.23
  }
}
```

## 11. Безопасность

### 11.1 Управление секретами
- Все API-ключи хранятся в `.env` файле или в системе управления секретами (Vault / Docker Secrets)
- Ни один ключ не попадает в код или репозиторий
- Ротация ключей — через переменные окружения без перезапуска контейнеров

### 11.2 Доступ к системе
- Telegram бот доступен только зарегистрированным пользователям (whitelist по `user_id`)
- Внутренние сервисы не открыты наружу — только через внутреннюю Docker-сеть
- Admin-панель (если будет) — только через VPN или IP-whitelist
