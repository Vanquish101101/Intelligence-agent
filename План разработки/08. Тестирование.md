## 1. Принципы тестирования

| Принцип | Что это означает |
|---|---|
| **Тест реальной работой** | Главный тест — агент ответил правильно на реальный запрос |
| **Тест fallback** | Нужно специально отключить инструмент и убедиться что система переключается |
| **Тест расходов** | Каждый тест замеряет стоимость вызовов API |
| **Тест скорости** | Фиксируем время от запроса до ответа |
| **Тест на граничных случаях** | Пустые результаты, недоступные платформы, некорректный ввод |

### 2.2 Уровень 2 — Integration: тест связки инструментов

Цель: убедиться что инструменты правильно передают данные друг другу.

**Цепочка сбора данных:**
- [ ] Apify собирает данные → Perplexity проверяет факты → результат объединён корректно
- [ ] Apify находит видео → Whisper транскрибирует → текст доступен для анализа
- [ ] Firecrawl парсит статью → GPT анализирует → хуки извлечены и понятны

**Цепочка оркестратора:**
- [ ] Оркестратор получает задачу → выбирает правильные инструменты по `registry.yaml`
- [ ] Оркестратор логирует каждый шаг → логи читаемы и полны
- [ ] Оркестратор замеряет стоимость → `meta.cost_usd` корректен

**Fallback-цепочка:**
- [ ] Apify недоступен → оркестратор переключается на Firecrawl
- [ ] Perplexity недоступен → отчёт идёт без проверки фактов (с пометкой)
- [ ] Whisper недоступен → видео пропускаются (с пометкой в отчёте)

### 2.3 Уровень 3 — End-to-End: полный цикл агента

Цель: проверить что пользователь получает качественный отчёт от начала до конца.

**Сценарий 1 — Ручной запуск через Telegram:**
- [ ] Пользователь пишет `/report` → агент отвечает "Работаю..." → через N минут приходит отчёт
- [ ] Отчёт содержит тренды, объяснение почему залетело, идеи для контента
- [ ] Время ответа укладывается в ожидаемый диапазон (< 5 мин для стандартного)

**Сценарий 2 — Поиск по теме:**
- [ ] Пользователь пишет `/search крипта` → агент ищет именно по этой теме
- [ ] Результаты релевантны теме, нет мусора не по теме

**Сценарий 3 — Автоотчёт по расписанию:**
- [ ] В настроенное время агент запускается без участия пользователя
- [ ] Отчёт приходит в Telegram без ошибок
- [ ] Если запуск по расписанию пропущен — пользователь уведомлён

**Сценарий 4 — Частичный сбой:**
- [ ] Одна из платформ (например TikTok) недоступна → отчёт приходит без неё, с пометкой
- [ ] Все платформы вернули пустые данные → агент сообщает об этом, не зависает

## 3. Качество отчёта — критерии оценки

Каждый отчёт проверяется по следующим параметрам:

| Критерий | Что проверяем | Проходит если |
|---|---|---|
| **Полнота** | Есть ли все секции (тренды, платформы, идеи) | Все секции присутствуют |
| **Релевантность** | Соответствуют ли тренды запрошенной теме | Более 80% контента по теме |
| **Глубина** | Объясняет ли агент ПОЧЕМУ контент залетел | Есть анализ хука/триггера |
| **Читаемость** | Легко ли читать отчёт в Telegram | Форматирование корректное |
| **Источники** | Есть ли ссылки или пояснения об источниках | Хотя бы 1 источник на тренд |
| **Скорость** | Время от запроса до получения | < 5 мин (стандарт), < 2 мин (быстро) |
| **Стоимость** | Расход на один отчёт | < $0.50 (MVP цель) |

## 5. Тестирование fallback и отказоустойчивости

### 5.1 Сценарии отказа

| Что ломаем | Как проверить | Ожидаемый результат |
|---|---|---|
| Apify недоступен | Остановить контейнер mcp-apify | Fallback на альтернативный парсер |
| Perplexity недоступен | Неверный API-ключ в тесте | Отчёт без раздела проверки фактов |
| Whisper недоступен | Остановить контейнер mcp-whisper | Видео пропущены, пометка в отчёте |
| GPT 5.5 недоступен | Симулировать timeout | Fallback на другую модель через OpenRouter |
| Redis недоступен | Остановить redis контейнер | Кэш не работает, но агент продолжает |
| Supabase недоступен | Неверный ключ | Отчёт отправляется, но не сохраняется в историю |
